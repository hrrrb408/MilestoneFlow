## 1. 文档说明

### 1.1 文档目的

本文档描述 MilestoneFlow MVP 的产品目标、用户、场景、流程、功能、页面、业务规则、优先级、数据指标和验收口径。

### 1.2 产品范围

MilestoneFlow 面向自由职业者和小型工作室，聚焦：

- 项目范围与报价确认；
- 里程碑交付与客户验收；
- 需求变更确认；
- 应收款跟踪；
- 自动提醒；
- AI 文本辅助；
- 审计记录。

### 1.3 版本定义

| 版本 | 定义 |
|---|---|
| MVP V1.0 | 完成最小可销售业务闭环 |
| V1.1 | 优化模板、导出、提醒和易用性 |
| V1.5 | 团队协作、基础报表、更多集成 |
| V2.0 | 支付、开放 API、多语言等扩展 |

---

## 2. 产品愿景

让项目制服务提供者用一个简单系统完成从报价到回款的全过程，并避免因为需求范围不清、验收无记录和催款遗漏造成损失。

---

## 3. 目标用户

### 3.1 用户画像 A：独立开发者

- 每年承接多个项目；
- 通常使用聊天工具、文档、表格管理项目；
- 经常遇到客户临时加需求；
- 最大痛点是边界不清、验收拖延和尾款难收；
- 愿意为省时间和减少损失付费。

### 3.2 用户画像 B：小型工作室负责人

- 团队规模约 2～10 人；
- 同时管理多个客户项目；
- 需要统一报价模板和交付流程；
- 需要知道哪些项目待验收、哪些款项已逾期；
- 需要操作记录以减少内部和客户争议。

### 3.3 用户画像 C：客户联系人

- 不一定愿意注册新系统；
- 只希望快速查看报价、交付内容或变更；
- 需要明确“确认后会发生什么”；
- 更偏好手机浏览器操作；
- 需要通过简单链接完成确认或反馈。

---

## 4. 用户痛点

| 编号 | 痛点 | 影响 |
|---|---|---|
| PAIN-01 | 项目范围散落在聊天记录 | 容易产生范围争议 |
| PAIN-02 | 报价版本不清晰 | 客户和服务方看到的版本不同 |
| PAIN-03 | 客户口头确认 | 缺少可追溯证据 |
| PAIN-04 | 临时需求未单独报价 | 服务方免费加班 |
| PAIN-05 | 交付后客户迟迟不验收 | 项目和回款停滞 |
| PAIN-06 | 应收款依靠人工记忆 | 漏催、晚催 |
| PAIN-07 | 工具过多 | 状态分散、重复录入 |
| PAIN-08 | 催款措辞困难 | 影响客户关系 |
| PAIN-09 | AI 输出不可控 | 可能生成错误条款或金额 |
| PAIN-10 | 客户不愿注册 | 确认流程中断 |

## 5. 产品目标与非目标

### 5.1 产品目标

1. 降低从创建项目到发送报价的时间。
2. 提升客户确认率。
3. 提升阶段验收完成率。
4. 降低未记录需求变更的比例。
5. 降低逾期款遗漏提醒的比例。
6. 形成可追溯的项目商业记录。

### 5.2 非目标

MilestoneFlow MVP 不是：

- 通用任务管理工具；
- 完整 CRM；
- 财务会计软件；
- 税务系统；
- 在线支付平台；
- 法律合同平台；
- 企业 ERP；
- 即时聊天软件。

---

## 6. 产品原则

1. 一个页面只解决一个主要任务。
2. 客户操作最多三步完成。
3. AI 输出必须可编辑。
4. 金额、日期、范围和状态必须明确。
5. 已确认版本不可直接覆盖。
6. 所有重要状态变化必须可追溯。
7. 系统失败时应允许用户复制链接或手动继续流程。
8. 首版优先桌面端管理、移动端客户确认。

## 7. 角色与权限

### 7.1 内部角色

| 角色 | 权限概述 |
|---|---|
| Owner | 工作空间全部权限、订阅、成员与删除 |
| Admin | 客户、项目、报价、验收、应收、模板和成员管理 |
| Member | 被授权项目的日常编辑、交付和查看 |
| Viewer | 只读查看内部项目和应收信息 |

### 7.2 外部客户

客户无需注册，通过安全公开链接执行：

- 查看报价；
- 确认、拒绝或留言；
- 查看交付；
- 验收、请求修改或拒绝；
- 查看变更单；
- 确认、拒绝或留言。

客户不可访问工作空间其他项目或内部信息。

## 9. 核心用户旅程

### 9.1 用户旅程一：发送报价

1. 用户注册并创建工作空间。
2. 创建客户。
3. 创建项目。
4. 输入需求描述。
5. 可选择 AI 整理为结构化范围。
6. 用户编辑工作范围、非工作范围、交付物和价格。
7. 保存报价草稿。
8. 发布报价版本。
9. 系统生成公开链接并发送邮件。
10. 客户查看并确认。
11. 项目进入“已确认/执行中”。

成功标准：

- 用户不需要离开系统即可完成报价发送；
- 客户无需注册；
- 已确认版本不可被直接修改。

### 9.2 用户旅程二：阶段交付与验收

1. 用户打开项目里程碑。
2. 上传文件或填写交付链接。
3. 填写交付说明。
4. 发送验收请求。
5. 客户查看并选择通过或请求修改。
6. 系统更新里程碑状态。
7. 通过后激活对应应收款。

成功标准：

- 验收结论有时间记录；
- 客户反馈与交付版本关联；
- 重复提交不会覆盖历史记录。

### 9.3 用户旅程三：需求变更

1. 用户收到客户新增需求。
2. 创建变更单。
3. 填写新增内容、费用、工期影响。
4. 发送客户确认。
5. 客户确认。
6. 系统更新项目总金额、交付日期和变更历史。

成功标准：

- 变更前后可对比；
- 未确认的变更不改变项目基线；
- 被拒绝变更可保留记录。

### 9.4 用户旅程四：应收跟踪

1. 报价或里程碑产生应收记录。
2. 系统在到期前提醒。
3. 用户可查看未付款、部分付款和已逾期列表。
4. 用户记录付款。
5. 系统更新项目与客户应收汇总。

成功标准：

- 不漏掉到期和逾期应收；
- 部分付款计算正确；
- 记录付款不等同于真实资金结算。

## 11. 功能需求详述

## 11.1 账号与认证

### 11.1.1 注册

用户输入：

- 邮箱；
- 密码；
- 昵称；
- 同意服务条款。

规则：

- 邮箱唯一；
- 密码满足最低复杂度；
- 注册后发送验证邮件；
- 未验证邮箱可登录，但关键发送功能受限；
- 防止重复发送验证邮件。

验收标准：

- 合法邮箱可注册；
- 重复邮箱不能注册；
- 弱密码得到明确提示；
- 验证链接过期后可重新发送。

### 11.1.2 登录

- 邮箱密码登录；
- 记住登录状态；
- 错误次数限制；
- 退出登录；
- 刷新令牌轮换；
- 异常登录记录。

### 11.1.3 密码重置

- 请求重置；
- 邮件链接；
- 链接有效期；
- 重置后旧会话失效。

## 11.5 范围与报价

### 11.5.1 报价内容

- 标题；
- 版本号；
- 项目背景；
- 工作范围；
- 非工作范围；
- 交付物；
- 时间计划；
- 修改次数；
- 客户责任；
- 报价明细；
- 税费说明；
- 总金额；
- 付款计划；
- 有效期；
- 备注。

### 11.5.2 报价状态

```text
草稿
已发布
已发送
已查看
已确认
已拒绝
已过期
已撤销
```

### 11.5.3 版本规则

- 每次发布产生不可变版本；
- 草稿可编辑；
- 已确认版本不可修改；
- 新修改必须创建新版本；
- 同一时刻仅允许一个有效待确认版本；
- 历史版本可查看但不可重新确认。

### 11.5.4 客户操作

客户可以：

- 查看完整报价；
- 查看有效期；
- 确认；
- 拒绝；
- 留言；
- 下载或打印；
- 查看确认结果。

确认前需勾选：

> 我已阅读并确认本报价所列项目范围、交付内容、付款计划与时间安排。

### 11.5.5 报价确认结果

确认后：

- 记录确认时间；
- 记录访问标识；
- 更新项目状态；
- 生成付款计划对应应收草稿；
- 向内部用户发送通知；
- 客户看到确认成功页。

### 11.6.4 客户验收

客户可选择：

- 通过验收；

规则：

- 请求修改必须填写原因；
- 拒绝验收必须填写原因；
- 已验收版本不可再次修改结论；
- 新交付产生新版本；
- 每个结论与具体交付版本关联。

- 标题；
- 原因；
- 新增内容；
- 修改内容；
- 删除内容；

- 费用变化；
- 工期变化；
- 新交付日期；
- 付款计划变化；

- 状态。

### 11.7.2 变更状态

```text
草稿
已发送
已查看
已确认
已拒绝
已撤销

```

### 11.7.3 业务规则

- 未确认变更不更新项目基线；

- 变更确认后可生成新的应收记录；
- 被拒绝的变更保留但不生效。

- 应收编号；
- 客户；
- 项目；
- 来源；
- 标题；
- 应收金额；
- 已收金额；
- 未收金额；
- 币种；
- 开票日期（可选）；
- 到期日期；
- 状态；
- 备注。

### 11.8.2 应收状态

```text
草稿
待付款
部分付款
已付款
已逾期
已取消
```

### 11.8.3 付款记录

字段：

- 付款金额；
- 付款日期；
- 付款方式；
- 交易参考号；
- 备注；
- 附件；
- 记录人。

规则：

- 付款金额必须大于零；
- 累计付款不得无提示超过应收金额；
- 允许超额付款，但必须二次确认；
- 全额支付后状态变为已付款；
- 未全额支付且到期后为已逾期或部分付款；
- 删除付款记录需高权限并写审计日志。

## 11.9 提醒与通知

### 11.9.1 提醒类型

- 报价已发送；
- 报价即将过期；
- 客户已查看报价；
- 客户已确认/拒绝报价；
- 里程碑待验收；
- 客户请求修改；
- 变更单已确认/拒绝；
- 应收到期前提醒；
- 到期日提醒；
- 逾期提醒；
- 邮件发送失败。

### 11.9.2 默认提醒规则

- 到期前 3 天；
- 到期当天；
- 逾期后第 3 天；
- 逾期后第 7 天。

用户可在发送前预览催款内容。MVP 不自动连续高频催款。

### 11.9.3 邮件失败处理

- 最多重试；
- 记录失败原因；
- 用户可手动重发；
- 提供复制公开链接；
- 不因邮件失败回滚业务发布动作。

---

## 11.10 AI 辅助

### 11.10.1 AI 范围整理

输入：

- 用户自然语言需求；
- 行业；
- 预算和工期（可选）。

输出草案：

- 项目背景；
- 工作范围；
- 非工作范围；
- 交付物；
- 里程碑；
- 风险提醒；
- 待确认问题。

### 11.10.2 AI 变更识别

输入：

- 原确认范围；
- 客户新增描述。

输出草案：

- 可能超范围项；
- 与原范围的差异；
- 建议新增交付物；
- 建议费用和时间影响描述；
- 风险说明。

AI 不得自动决定最终金额。

### 11.10.3 AI 催款文案

可选择语气：

- 温和提醒；
- 正式提醒；
- 逾期提醒；
- 最终提醒。

输出必须包含：

- 项目名称；
- 应收金额；
- 到期日；
- 付款状态；
- 联系方式；
- 不得使用威胁性或虚假法律措辞。

### 11.10.4 AI 通用规则

- 输出仅为草案；
- 用户编辑确认后保存；
- 调用失败可重试；
- 调用超时不阻塞手动流程；
- 记录使用次数和错误，不记录不必要的敏感全文；
- 支持关闭 AI 功能。

- 登录与安全事件；
- 工作空间设置；
- 成员与角色；
- 客户变更；
- 报价发布、撤销和确认；
- 交付与验收；
- 变更确认；
- 付款记录；
- 公开链接创建、撤销和使用；
- 数据导出；
- 高风险删除。

日志字段：

- 操作者；

- IP 或来源标识；

## 13. 状态与异常设计

### 13.1 公开链接异常

| 场景 | 页面行为 |
|---|---|
| 链接不存在 | 显示无效链接，不泄露对象信息 |
| 链接已过期 | 显示已过期和联系提示 |
| 链接已撤销 | 显示已撤销 |
| 对象已确认 | 显示结果只读页 |
| 对象已更新 | 提示使用最新链接 |
| 网络失败 | 保留输入并允许重试 |
| 重复提交 | 返回已有结果，保证幂等 |

### 13.2 邮件异常

- 发布成功但邮件失败：业务对象仍保持已发布；
- 展示失败提示和重发按钮；
- 记录失败原因；
- 提供复制链接。

### 13.3 AI 异常

- 超时；
- 限流；
- 返回格式错误；
- 内容为空；
- 服务不可用。

所有情况均应允许用户继续手动编辑。

## 14. 业务规则

1. 所有金额使用工作空间默认币种，MVP 不做自动汇率换算。
2. 已确认报价、验收和变更不可直接覆盖。
3. 项目总金额由已确认报价和已确认变更计算。
4. 未确认草稿不得影响应收统计。
5. 所有日期按工作空间时区展示和计算。
6. 客户确认链接只能访问一个明确业务对象。
7. 公开链接令牌可撤销、可过期。
8. 已归档客户不得创建新项目，恢复后才可使用。
9. 已归档项目不允许新增交付、变更或应收。
10. 记录付款只代表用户登记，不代表平台实际收款。
11. AI 不得自动发送邮件或自动确认业务对象。
12. 操作者必须有当前工作空间权限。
13. 数据查询必须带工作空间边界。
14. 删除关键业务对象优先采用归档或软删除。
15. 确认按钮需要防重复提交。

## 16. 非功能产品要求

1. 管理端支持常见桌面分辨率。
2. 客户公开页支持手机浏览器。
3. 重要按钮有加载、防重复提交和结果反馈。
4. 用户输入在网络异常后尽量不丢失。
5. 列表支持分页、搜索和基础筛选。
6. 关键页面需有空状态和新手引导。
7. 邮件内容需清楚说明发送方和操作目的。
8. 客户链接页面不得展示内部备注。
9. 所有时间显示使用工作空间时区。
10. 金额显示遵循币种格式。

## 17. MVP 发布验收

MVP 上线必须满足：

- 用户可完成注册、登录和邮箱验证；
- 用户可创建工作空间、客户和项目；
- 用户可发布报价并由客户确认；
- 用户可提交里程碑并由客户验收；
- 用户可创建变更单并由客户确认；
- 用户可管理应收和记录部分/全部付款；
- 系统能识别并展示逾期；
- 系统能发送关键邮件；
- AI 不可用时核心流程仍可完成；
- 公开链接具备过期、撤销和幂等能力；
- 核心业务操作具有审计记录；
- 无 P0/P1 阻断缺陷；
- 备份和恢复流程验证通过；
- 隐私政策、服务条款和基础帮助页可访问。

## 19. 产品决策记录

| 决策 | 结论 | 原因 |
|---|---|---|
| 客户是否注册 | 不要求 | 降低确认阻力 |
| 首版是否收款 | 不处理真实资金 | 降低合规与开发复杂度 |
| AI 是否自动执行 | 否 | 避免错误和失控 |
| 是否做通用任务管理 | 否 | 保持垂直定位 |
| 是否做移动 App | 否 | 优先 Web 验证 |
| 是否支持多币种换算 | 否 | 避免财务复杂度 |
| 是否允许覆盖已确认版本 | 否 | 保证可追溯 |
| 是否支持公开链接撤销 | 是 | 满足安全需要 |

## 20. PRD 结论

MilestoneFlow MVP 应围绕一个可完整演示和真实使用的商业闭环交付，而不是堆叠通用管理功能。所有开发优先级必须以“是否帮助用户明确范围、完成验收或收回款项”为判断标准。
