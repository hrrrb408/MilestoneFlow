**状态：** 软件需求基线  
**编制日期：** 2026-06-06

## 1. 引言

### 1.1 目的

本文档定义 MilestoneFlow MVP 的软件功能需求、外部接口需求、数据需求、非功能需求、业务规则、约束、验收标准和需求追踪关系。

本文档面向：

- 产品负责人；
- 架构师；
- 前端与后端开发人员；
- 测试人员；
- 运维人员；
- 安全评审人员；
- 项目负责人。

### 1.2 系统范围

MilestoneFlow 是一个多租户 SaaS Web 系统，为自由职业者和小型工作室提供项目范围、报价确认、里程碑交付、客户验收、需求变更、应收跟踪、提醒和审计能力。

### 1.3 术语

| 术语 | 定义 |
|---|---|
| 工作空间 | 一个独立的业务租户和数据边界 |
| 内部用户 | 登录系统并属于工作空间的成员 |
| 客户 | 接受报价、交付或变更的外部对象 |
| 公开链接 | 客户无需登录即可访问单一业务对象的安全链接 |
| 报价版本 | 一次发布后不可变的项目范围和商业条件快照 |
| 里程碑 | 项目阶段性工作和验收单元 |
| 交付版本 | 某次里程碑提交的不可变记录 |
| 变更单 | 对已确认范围、费用或工期的正式修改请求 |
| 应收款 | 用户记录的待收金额，不代表平台实际结算 |
| AI 草案 | 模型生成、必须经用户确认的辅助内容 |
| 审计日志 | 不允许普通用户修改的关键操作记录 |

### 1.4 设计原则

- 模块化单体；
- RESTful API；
- 多租户隔离；
- 版本不可变；
- 公开链接最小权限；
- 人工确认 AI 输出；
- 可测试、可观测、可恢复。

## 2. 总体描述

### 2.1 产品上下文

```text
Vue Web 管理端
Vue/响应式客户公开页
        ↓
Spring Boot REST API
        ↓
认证与权限
客户与项目
报价与确认
里程碑与验收
变更管理
应收与付款记录
通知与邮件
AI 适配器
文件适配器
审计与报表
        ↓
PostgreSQL
对象存储
邮件服务
GLM API
```

### 2.2 用户类别

| 用户类别 | 特征 |
|---|---|
| Owner | 创建工作空间，拥有全部权限 |
| Admin | 负责日常管理与成员管理 |
| Member | 负责项目编辑、交付和跟进 |
| Viewer | 只读访问 |
| Client | 通过公开链接执行确认或反馈 |
| System Job | 执行提醒、过期和重试任务 |

### 2.3 运行环境

#### 客户端

- Chrome、Edge、Safari、Firefox 的主流稳定版本；
- 管理端主要面向桌面浏览器；
- 客户公开页支持移动浏览器；
- JavaScript 必须启用。

#### 服务端

- Linux；
- Java 21；
- Spring Boot 3；
- PostgreSQL；
- Docker；
- Nginx；
- S3 兼容对象存储。

### 2.4 约束

1. MVP 使用单体模块化架构。
2. 金额计算必须使用 Decimal。
3. 已确认业务版本不可直接覆盖。
4. 客户无需注册。
5. 不处理真实资金。
6. AI 为可选辅助服务。
7. 系统必须支持 GLM 服务不可用时降级。
8. 所有租户业务数据必须通过 workspace_id 隔离。
9. 数据库变更必须通过迁移脚本。
10. 生产环境禁止使用默认密钥和开发配置。

### 2.5 假设与依赖

- 用户具有可用邮箱；
- 工作空间至少有一个 Owner；
- 邮件服务提供商可用；
- 对象存储服务可用；
- GLM API 可通过 HTTPS 调用；
- 系统时间由可靠时间源同步；
- 用户对录入的客户数据和金额负责。

## 3. 系统功能需求

## 3.1 认证与账号

### FR-AUTH-001 用户注册

系统应允许用户使用邮箱、密码和昵称注册账号。

验收条件：

- 邮箱格式合法；
- 邮箱全局唯一；
- 密码符合策略；
- 注册成功后创建未验证账号；
- 发送验证邮件任务。

### FR-AUTH-002 邮箱验证

系统应允许用户通过一次性令牌验证邮箱。

规则：

- 令牌有有效期；
- 令牌只能使用一次；
- 验证成功后记录 verified_at；
- 重复访问返回已验证结果。

### FR-AUTH-003 登录

系统应验证邮箱和密码并签发 Access Token 与 Refresh Token。

### FR-AUTH-004 刷新会话

系统应支持 Refresh Token 轮换，并使已使用的旧令牌失效。

### FR-AUTH-005 退出登录

系统应允许当前会话退出，并撤销对应刷新令牌。

### FR-AUTH-006 密码重置

系统应通过邮箱重置令牌完成密码重置。

### FR-AUTH-007 登录保护

系统应限制连续失败登录，并记录安全事件。

### FR-AUTH-008 会话失效

密码重置、账号禁用或高风险安全操作后，系统应使相关会话失效。

## 3.2 工作空间与成员

### FR-WS-001 创建工作空间

注册用户应能创建工作空间，并自动成为 Owner。

### FR-WS-002 工作空间设置

Owner 或 Admin 应能修改名称、Logo、展示名称、时区、日期格式、默认币种和联系邮箱。

### FR-WS-003 邀请成员

Owner 或 Admin 应能通过邮箱邀请成员，并指定角色。

### FR-WS-004 接受邀请

被邀请用户登录后可接受有效邀请并加入工作空间。

### FR-WS-005 角色管理

Owner 或 Admin 应能调整成员角色，但不得移除最后一个 Owner。

### FR-WS-006 移除成员

授权用户应能移除成员，并使其立即失去工作空间访问权限。

### FR-WS-007 租户隔离

系统对任何业务查询和写入必须校验当前用户属于对应工作空间。

### FR-WS-008 工作空间切换

属于多个工作空间的用户应能切换当前工作空间。

---

## 3.3 客户管理

### FR-CLIENT-001 创建客户

授权用户应能创建个人或公司客户。

### FR-CLIENT-002 编辑客户

授权用户应能编辑客户基础信息和联系人。

### FR-CLIENT-003 查询客户

系统应支持按名称、联系人、邮箱和标签搜索。

### FR-CLIENT-004 客户详情

系统应展示客户基础信息、关联项目、待确认事项和应收汇总。

### FR-CLIENT-005 归档客户

授权用户应能归档客户。归档客户不可创建新项目。

### FR-CLIENT-006 恢复客户

授权用户应能恢复归档客户。

### FR-CLIENT-007 客户删除限制

存在项目、报价、应收或审计记录的客户不得物理删除。

---

## 3.4 项目管理

### FR-PROJ-001 创建项目

授权用户应能为活跃客户创建项目。

### FR-PROJ-002 编辑项目

授权用户应能编辑未归档项目的可变字段。

### FR-PROJ-003 项目状态

系统应支持：DRAFT、PENDING_QUOTE_CONFIRMATION、CONFIRMED、IN_PROGRESS、PENDING_ACCEPTANCE、COMPLETED、PAUSED、CANCELLED、ARCHIVED。

### FR-PROJ-004 状态转换校验

系统应按照状态机校验项目状态转换，不允许任意跳转。

### FR-PROJ-005 项目概览

系统应计算并展示总金额、已收、待收、下一里程碑和待处理事项。

### FR-PROJ-006 归档项目

归档项目应进入只读状态，不得新增报价、交付、变更或应收。

### FR-PROJ-007 项目访问权限

Member 只能访问被授权项目；Owner 和 Admin 可访问工作空间全部项目。

## 3.5 报价与范围

### FR-QUOTE-001 创建报价草稿

系统应允许用户创建报价草稿，包含背景、工作范围、非工作范围、交付物、时间计划、修改次数、客户责任、报价明细、付款计划和有效期。

### FR-QUOTE-002 保存草稿

草稿可多次保存，保存时不创建公开链接。

### FR-QUOTE-003 报价校验

发布前系统必须校验：

- 客户存在且活跃；
- 至少一项工作范围；
- 总金额合法；
- 付款计划金额合计符合规则；
- 有效期合法；
- 币种存在。

### FR-QUOTE-004 发布报价版本

发布应创建不可变报价版本，并分配递增版本号。

### FR-QUOTE-005 单一有效版本

同一项目同一时刻只允许一个有效的待确认报价版本。

### FR-QUOTE-006 公开链接

系统应为报价版本生成高熵随机令牌，并存储令牌哈希。

### FR-QUOTE-007 发送报价

系统应支持发送报价邮件，或复制公开链接。

### FR-QUOTE-008 报价查看记录

客户首次打开报价时，系统应记录查看时间和安全来源信息。

### FR-QUOTE-009 客户确认报价

客户应能确认有效报价版本。系统必须保证该操作幂等。

### FR-QUOTE-010 客户拒绝报价

客户应能拒绝有效报价并可填写原因。

### FR-QUOTE-011 客户留言

客户可在确认或拒绝前提交留言。

### FR-QUOTE-012 报价过期

到达有效期后，未确认报价应自动变为已过期。

### FR-QUOTE-013 撤销报价

授权用户应能撤销未确认报价，撤销后公开链接不可继续操作。

### FR-QUOTE-014 报价确认结果

报价确认后系统应：

- 记录确认；
- 更新项目状态；
- 创建付款计划对应应收草稿；
- 发送内部通知；
- 写入审计日志。

### FR-QUOTE-015 新报价版本

对已发布报价的修改必须创建新草稿并再次发布，不得覆盖旧版本。

## 3.6 里程碑与交付

### FR-MS-001 创建里程碑

用户应能为未归档项目创建里程碑。

### FR-MS-002 里程碑排序

系统应支持里程碑顺序调整，并保证排序稳定。

### FR-MS-003 里程碑状态

系统应支持 NOT_STARTED、IN_PROGRESS、PENDING_ACCEPTANCE、REVISION_REQUIRED、ACCEPTED、CANCELLED。

### FR-MS-004 提交交付版本

用户应能创建交付版本，包含说明、文件和外部链接。

### FR-MS-005 交付版本不可变

提交后的交付版本不可直接修改；再次提交需创建新版本。

### FR-MS-006 发送验收请求

系统应生成里程碑验收公开链接并发送邮件或允许复制。

### FR-MS-007 客户通过验收

客户应能通过当前有效交付版本。

### FR-MS-008 客户请求修改

客户请求修改时必须填写反馈。

### FR-MS-009 客户拒绝验收

客户拒绝时必须填写原因。

### FR-MS-010 验收幂等

重复提交同一验收结果不得生成重复记录。

### FR-MS-011 验收后处理

验收通过后系统应：

- 标记里程碑已验收；
- 记录验收时间；
- 激活或创建关联应收；
- 更新项目进度；
- 写入通知和审计。

### FR-MS-012 历史版本

内部用户应能查看所有交付版本和对应客户反馈。

## 3.7 需求变更

### FR-CR-001 创建变更草稿

授权用户应能创建关联项目和报价版本的变更单。

### FR-CR-002 变更内容

变更单应支持新增、修改、删除、不受影响内容、费用变化、工期变化和付款计划变化。

### FR-CR-003 发布变更版本

发布后应形成不可变变更版本和公开链接。

### FR-CR-004 客户查看变更

客户应能查看变更前提、变更内容、金额和工期影响。

### FR-CR-005 客户确认变更

客户确认后系统应更新项目商业基线。

### FR-CR-006 客户拒绝变更

客户可拒绝变更并填写原因。

### FR-CR-007 未确认不生效

草稿、已发送、已查看状态的变更不得影响项目金额、日期或应收。

### FR-CR-008 确认后应收

确认后可根据付款计划创建或调整应收记录。

### FR-CR-009 撤销与过期

未确认变更可撤销，并可设置有效期。

### FR-CR-010 版本追溯

系统应保留变更前后的项目商业基线。

---

## 3.8 应收与付款记录

### FR-REC-001 创建应收

系统应允许由报价、里程碑、变更或手动方式创建应收。

### FR-REC-002 应收金额

应收金额必须大于零，币种必须明确。

### FR-REC-003 应收状态

系统应支持 DRAFT、PENDING、PARTIALLY_PAID、PAID、OVERDUE、CANCELLED。

### FR-REC-004 记录付款

授权用户应能记录付款金额、日期、方式、参考号、备注和附件。

### FR-REC-005 金额计算

系统应基于付款记录计算已收金额和未收金额。

### FR-REC-006 部分付款

已收金额大于零且小于应收金额时，状态应为部分付款；若已过期，UI 应同时展示逾期标识。

### FR-REC-007 全额付款

已收金额等于应收金额时，状态应为已付款。

### FR-REC-008 超额付款

超过应收金额时，系统必须要求二次确认并记录超额金额。

### FR-REC-009 逾期计算

到期日结束后仍未付清的应收，应标记为逾期。

### FR-REC-010 取消应收

授权用户可取消未付清应收，但必须填写原因。

### FR-REC-011 删除付款记录

仅 Owner 或 Admin 可删除错误付款记录，删除操作必须审计。

### FR-REC-012 汇总

系统应按工作空间、客户和项目汇总应收、已收和未收金额。

### FR-REC-013 免责声明

系统界面应明确付款记录为用户手工登记，不代表平台实际资金结算。

## 3.9 通知与邮件

### FR-NOTIF-001 站内通知

系统应为重要客户动作生成站内通知。

### FR-NOTIF-002 邮件任务

邮件发送应通过异步任务处理。

### FR-NOTIF-003 邮件状态

系统应记录待发送、发送中、成功、失败和取消状态。

### FR-NOTIF-004 邮件重试

临时失败应按配置重试，永久失败不应无限重试。

### FR-NOTIF-005 手动重发

授权用户应能手动重发失败邮件。

### FR-NOTIF-006 默认提醒

系统应支持到期前、到期日和逾期后的默认提醒。

### FR-NOTIF-007 防重复提醒

同一应收同一提醒节点不得重复创建相同邮件任务。

### FR-NOTIF-008 通知偏好

工作空间应能配置非关键提醒；关键业务通知不得被全部关闭。

### FR-NOTIF-009 公开链接备用

邮件失败后，系统仍应允许复制公开链接。

## 3.10 AI 辅助

### FR-AI-001 AI Provider 接口

系统应通过统一接口调用 GLM，不在业务模块中直接耦合供应商 SDK。

### FR-AI-002 范围整理

用户可请求将自然语言整理为结构化范围草案。

### FR-AI-003 变更整理

用户可请求比较已确认范围和新增要求，输出差异草案。

### FR-AI-004 催款草案

用户可请求生成不同语气的催款文案。

### FR-AI-005 人工确认

AI 输出不得自动保存为正式版本或自动发送。

### FR-AI-006 超时降级

AI 超时或失败时，系统应返回可理解错误，不影响手动流程。

### FR-AI-007 使用限额

系统应支持按套餐限制 AI 调用次数。

### FR-AI-008 内容安全

系统不得向 AI 发送与任务无关的敏感字段。

### FR-AI-009 输出校验

结构化 AI 输出必须进行 Schema 校验；不合格输出作为纯文本草案处理或要求重试。

### FR-AI-010 AI 关闭

工作空间应可关闭 AI 功能。

---

## 3.11 文件管理

### FR-FILE-001 文件上传

用户应能上传报价附件、交付物、付款凭证和工作空间 Logo。

### FR-FILE-002 文件限制

系统应限制文件大小、类型和数量。

### FR-FILE-003 文件存储

文件应存储于对象存储，数据库保存元数据。

### FR-FILE-004 文件授权

下载文件时必须校验工作空间权限或公开链接权限。

### FR-FILE-005 文件扫描

生产环境应预留恶意文件扫描能力；MVP 至少进行扩展名、MIME 和大小校验。

### FR-FILE-006 文件删除

被历史版本引用的文件不得直接物理删除。

### FR-FILE-007 临时访问

私有文件下载应使用短时有效访问地址或受控转发。

---

## 3.12 审计

### FR-AUDIT-001 审计对象

系统应记录认证、权限、报价、验收、变更、付款、公开链接和高风险删除操作。

### FR-AUDIT-002 审计不可变

普通业务接口不得修改或删除审计日志。

### FR-AUDIT-003 审计查询

Owner 和 Admin 应能按对象、操作者、时间和操作类型查询日志。

### FR-AUDIT-004 敏感数据

审计日志不得记录密码、完整令牌、完整银行卡信息或 AI 密钥。

### FR-AUDIT-005 请求关联

审计日志应包含 request_id 或 correlation_id。

---

## 3.13 公开链接

### FR-PUBLIC-001 链接生成

每个公开链接应使用高熵随机令牌，数据库仅保存哈希。

### FR-PUBLIC-002 最小权限

链接只能访问其关联的单一报价、交付或变更对象。

### FR-PUBLIC-003 有效期

链接应支持有效期。

### FR-PUBLIC-004 撤销

## 7. 业务状态机

## 7.1 报价状态机

```text
DRAFT
  └─ publish → PUBLISHED/SENT
                    ├─ view → VIEWED
                    ├─ confirm → CONFIRMED
                    ├─ reject → REJECTED
                    ├─ expire → EXPIRED
                    └─ revoke → REVOKED
```

终态：CONFIRMED、REJECTED、EXPIRED、REVOKED。

## 7.2 里程碑状态机

```text
NOT_STARTED
  └─ start → IN_PROGRESS
                └─ submit → PENDING_ACCEPTANCE
                               ├─ accept → ACCEPTED
                               ├─ revision → REVISION_REQUIRED
                               │                  └─ resubmit → PENDING_ACCEPTANCE
                               └─ cancel → CANCELLED
```

## 7.3 变更单状态机

```text
DRAFT
  └─ publish → SENT
                  ├─ view → VIEWED
                  ├─ confirm → CONFIRMED
                  ├─ reject → REJECTED
                  ├─ expire → EXPIRED
                  └─ revoke → REVOKED
```

## 7.4 应收状态机

```text
DRAFT
  └─ activate → PENDING
                  ├─ partial payment → PARTIALLY_PAID
                  ├─ full payment → PAID
                  ├─ overdue → OVERDUE
                  └─ cancel → CANCELLED
```

PARTIALLY_PAID 到期后可保留状态并增加 overdue_flag，或进入 OVERDUE 并保留支付比例。实现方案需在详细设计中固定。

## 8. 用例

## UC-001 发布并确认报价

**参与者：** 内部用户、客户、邮件服务  
**前置条件：** 用户已登录；项目和客户存在。

**主流程：**

1. 用户编辑报价草稿。
2. 系统校验内容。
3. 用户发布。
4. 系统创建报价版本和公开链接。
5. 系统创建邮件任务。
6. 客户打开链接。
7. 系统记录查看。
8. 客户确认。
9. 系统幂等写入确认记录。
10. 系统更新项目状态。
11. 系统创建应收草稿。
12. 系统发送内部通知。

**异常流程：**

- 报价已过期；
- 链接已撤销；
- 已存在新版本；
- 邮件失败；
- 重复确认；
- 客户网络中断。

**后置条件：** 报价版本已确认且不可变。

## UC-002 提交并验收里程碑

**参与者：** 内部用户、客户  
**前置条件：** 项目未归档，里程碑可提交。

**主流程：**

1. 用户创建交付版本。
2. 系统保存文件和说明。
3. 用户发送验收请求。
4. 客户查看。
5. 客户通过验收。
6. 系统记录验收。
7. 系统更新里程碑。
8. 系统激活应收。

## UC-003 确认需求变更

**参与者：** 内部用户、客户  
**前置条件：** 项目存在已确认报价。

**主流程：**

1. 用户创建变更草稿。
2. 填写范围、费用和工期影响。
3. 发布并发送。
4. 客户确认。
5. 系统更新项目商业基线。
6. 系统创建新应收。
7. 系统记录审计。

## UC-004 记录部分付款

**参与者：** 内部用户  
**前置条件：** 应收处于待付款、部分付款或逾期状态。

**主流程：**

1. 用户输入付款金额和日期。
2. 系统校验金额。
3. 系统创建付款记录。
4. 系统重新计算已收和未收。
5. 系统更新状态。
6. 系统写入审计。

## 14. SRS 验收结论

本 SRS 覆盖 MilestoneFlow MVP 的主要软件行为、接口、数据、性能、安全、可靠性、隐私、测试和发布要求。开发团队应将所有功能需求转化为设计项、开发任务和测试用例，并维护需求追踪关系。

任何改变以下内容的需求都必须执行正式变更评审：

- 客户无需注册；
- 不处理真实资金；
