# MilestoneFlow 项目立项文档

**文档编号：** MF-PROJ-001  
**版本：** V1.0  
**状态：** 立项基线  
**编制日期：** 2026-06-06  
**项目名称：** MilestoneFlow  
**项目类型：** 面向自由职业者与小型工作室的项目交付与回款管理 SaaS  
**目标版本：** MVP V1.0

---

## 1. 文档目的

本文档用于明确 MilestoneFlow 项目的立项依据、商业目标、建设范围、实施计划、资源投入、风险控制、验收标准与项目治理方式，为后续产品需求、软件需求、架构设计、开发、测试、部署和运营提供统一基线。

本项目采用“先验证、后扩展”的原则。MVP 不追求通用项目管理平台的完整功能，而是围绕以下核心闭环建设：

> 项目范围明确 → 报价确认 → 分阶段交付 → 客户验收 → 需求变更确认 → 应收款跟踪与催款。

---

## 2. 项目背景

自由职业者、小型软件工作室、设计工作室、视频工作室、营销团队和咨询顾问通常以项目制方式开展业务。在实际合作中，普遍存在以下问题：

1. 项目范围只存在于聊天记录中，缺少结构化确认。
2. 客户临时增加需求，双方难以判断是否属于原范围。
3. 阶段成果已经交付，但客户迟迟不确认验收。
4. 报价、变更、交付、验收和收款信息分散在多个工具中。
5. 付款到期后依赖人工提醒，容易漏催或催款措辞不当。
6. 发生争议时，缺少完整、可追溯的版本记录和操作证据。
7. 通用项目管理工具重任务协作，轻客户确认与商业回款。

MilestoneFlow 将以项目商业闭环为核心，而不是以内部任务管理为核心。

---

## 3. 项目愿景与定位

### 3.1 项目愿景

成为自由职业者和小型工作室管理项目交付、客户确认与回款流程的轻量化基础工具。

### 3.2 产品定位

MilestoneFlow 是一款面向项目制服务提供者的 SaaS 产品，帮助用户管理：

- 客户；
- 项目范围与报价；
- 项目里程碑；
- 交付记录与客户验收；
- 需求变更；
- 应收款与付款状态；
- 自动提醒；
- 全流程操作记录。

### 3.3 一句话价值主张

> 减少免费加需求，保留验收证据，更快收回项目款。

### 3.4 首批目标用户

- 独立软件开发者；
- 网站、小程序和系统开发工作室；
- UI、平面和品牌设计师；
- 摄影与视频制作工作室；
- 广告投放、社交媒体和内容代运营团队；
- 独立顾问与小型咨询团队。

## 4. 立项目标

### 4.1 商业目标

MVP 上线后 90 天内完成以下验证：

| 编号 | 目标 | 验证标准 |
|---|---|---|
| BG-01 | 验证核心痛点 | 至少访谈 20 名目标用户 |
| BG-02 | 验证真实使用 | 至少 10 名用户创建真实项目 |
| BG-03 | 验证闭环价值 | 至少 5 个项目完成客户确认或验收 |
| BG-04 | 验证付费意愿 | 至少获得 3 名付费用户或等值早鸟订单 |
| BG-05 | 验证留存 | 4 周内至少 30% 的试用用户再次使用 |
| BG-06 | 控制成本 | 单个活跃用户月均基础设施成本可控 |

以上指标是验证目标，不构成收益保证。

### 4.2 产品目标

1. 用户可以在 10 分钟内创建项目并发送报价确认链接。
2. 客户无需注册即可完成报价确认、阶段验收和变更确认。
3. 所有关键业务对象均保留版本、时间和操作者记录。
4. 用户可以清楚查看待确认、待验收、待付款和已逾期事项。
5. AI 只作为辅助能力，不成为主流程的单点依赖。
6. 核心流程支持桌面端和移动浏览器访问。

### 4.3 技术目标

1. 建立可维护的单体模块化架构。
2. 建立覆盖核心业务的自动化测试。
3. 使用 Docker Compose 完成开发和生产基础部署。
4. 支持结构化日志、审计日志、异常监控和数据库备份。
5. 核心接口具备清晰的权限校验、幂等控制和输入校验。
6. 为后续团队协作、支付接入和多语言预留扩展点。

- 客户历史付款状态。

- 创建里程碑；

- 创建变更单；

- 生成应收记录；

- 生成催款文案；

#### 5.1.8 审计与通知

- 关键操作审计日志；
- 邮件发送记录；
- 公开链接访问与操作记录；
- 站内通知；
- 失败重试与错误提示。

## 6. 核心业务流程

### 6.1 报价确认流程

1. 用户创建客户。
2. 用户创建项目。
3. 用户录入项目需求或使用 AI 整理需求。
4. 用户生成报价版本。
5. 系统生成公开确认链接。
6. 客户查看并确认、拒绝或留言。
7. 客户确认后，该版本成为项目商业基线。
8. 系统记录确认时间、来源和版本。

### 6.2 里程碑验收流程

1. 用户创建里程碑。
2. 用户提交交付物和说明。
3. 系统发送验收链接。
4. 客户查看交付内容。
5. 客户选择通过、请求修改或拒绝。
6. 系统记录验收结论和反馈。
7. 验收通过后，里程碑进入已验收状态。
8. 需要时生成或激活对应应收记录。

### 6.3 需求变更流程

1. 用户发现客户要求超出已确认范围。
2. 用户创建变更单。
3. AI 可辅助提取变更点、费用影响和时间影响。
4. 用户确认后发送客户。
5. 客户确认、拒绝或留言。
6. 确认后更新项目金额、交付日期或范围基线。
7. 系统保留变更前后版本和审计记录。

### 6.4 应收提醒流程

1. 系统根据付款计划创建应收记录。
2. 到期前发送提醒。
3. 到期未付则标记为待付款。
4. 逾期后标记为已逾期并触发提醒。
5. 用户可记录部分付款或全部付款。
6. 系统保留每次提醒和收款记录。

## 7. 产品原则

1. **商业闭环优先：** 功能必须服务于范围确认、交付、验收或回款。
2. **客户低门槛：** 客户无需注册即可完成关键确认动作。
3. **人工最终确认：** AI 不得自动发送报价、变更或催款。
4. **记录可追溯：** 关键业务对象必须保留版本和操作历史。
5. **默认安全：** 公开链接需具备随机令牌、有效期和撤销能力。
6. **小团队友好：** 控制配置复杂度，避免企业软件式重流程。
7. **先验证后扩展：** 未验证付费前，不建设重型平台能力。

## 9. 建议技术路线

### 9.1 技术栈

- 前端：Vue 3、TypeScript、Vite、Element Plus；
- 后端：Spring Boot 3、Java 21；
- 数据库：PostgreSQL；
- ORM：MyBatis-Plus 或 Spring Data JPA，项目启动前二选一；
- 文件存储：开发环境本地存储，生产环境 S3 兼容对象存储；
- 邮件：可替换的邮件服务适配层；
- AI：GLM API，通过统一 AI Provider 接口接入；
- 认证：JWT Access Token + Refresh Token；
- API：RESTful；
- 部署：Docker Compose + Nginx；
- 监控：结构化日志、健康检查、错误告警；
- 测试：JUnit 5、Testcontainers、Vitest、Playwright。

### 9.2 架构原则

- 单体模块化；
- 领域按模块隔离；
- 外部服务通过适配器封装；
- 公开链接流程与内部登录流程分离；
- AI 调用异步化或超时降级；
- 所有关键写操作具备审计记录；
- 数据库结构通过迁移脚本管理。

## 15. 主要风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|---|---|---|---|
| 用户认为表格已足够 | 高 | 高 | 突出确认证据、变更和回款闭环 |
| 功能范围膨胀 | 高 | 高 | 用 MVP 范围清单和变更评审控制 |
| 客户不愿注册 | 高 | 中 | 采用无需注册的安全公开链接 |
| 邮件进入垃圾箱 | 中 | 高 | 域名认证、重试、站内状态与复制链接 |
| AI 输出不准确 | 高 | 中 | 只生成草案，用户确认后保存 |
| 公开链接泄露 | 中 | 高 | 高强度令牌、有效期、撤销和访问日志 |
| 财务金额计算错误 | 低 | 高 | Decimal、统一舍入规则、自动化测试 |
| 数据丢失 | 低 | 高 | 自动备份、恢复演练和迁移审查 |
| 单人开发延期 | 高 | 中 | 控制功能、短迭代、优先核心闭环 |
| 付费意愿不足 | 中 | 高 | 开发前预售，未达阈值及时转向 |

## 16. 质量与安全要求

1. 金额字段禁止使用浮点数。
2. 公开令牌不得明文写入日志。
3. 密码必须使用成熟的单向哈希算法。
4. 所有工作空间数据必须进行租户隔离。
5. 文件上传需限制类型、大小和数量。
6. 关键业务操作必须写入审计日志。
7. AI 请求不得包含不必要的敏感数据。
8. 数据删除应采用软删除或归档，关键记录不得直接覆盖。
9. 上线前必须完成身份认证、越权、公开链接、上传和注入测试。
10. 数据库备份必须可恢复，而不是只验证备份文件存在。

---

## 17. 立项验收条件

项目进入正式开发前，需满足：

- 立项文档评审通过；
- PRD V1.0 评审通过；
- SRS V1.0 评审通过；
- 页面原型覆盖核心流程；
- 数据模型初稿完成；
- 技术栈冻结；
- MVP 范围冻结；
- 至少 5 名目标用户表示愿意试用；
- 至少 2 名目标用户愿意提供真实项目数据进行测试；
- 开发、测试和生产环境隔离方案明确。

---

## 18. 项目变更管理

任何新增需求必须记录：

- 需求来源；
- 商业价值；
- 用户影响；
- 开发成本；
- 测试成本；
- 上线风险；
- 是否影响 MVP 日期；
- 是否替换现有需求。

MVP 期间只有以下需求可进入 P0：

1. 阻断核心业务闭环；
2. 导致数据错误或安全风险；
3. 影响付费验证；
4. 影响法律或合规底线。

---

## 19. 立项结论

MilestoneFlow 具备明确的目标用户、可验证的商业痛点、可控制的技术复杂度和清晰的 MVP 闭环。项目适合采用单体模块化架构，由单人开发者配合 Claude Code + GLM 完成。

立项后应严格执行：

> 市场验证 → 需求基线 → 架构设计 → 增量开发 → 自动化测试 → 小范围试运营 → 付费验证 → 再决定扩展。

**立项建议：批准进入需求评审与原型验证阶段。**
