# 《MilestoneFlow Pilot MVP V0.1 索引和约束设计》

## 1. 设计目标

1. 数据库作为租户隔离、唯一性、金额合法性和不可变历史的最后防线。
2. 索引优先服务真实查询：所有业务列表首先按 `workspace_id` 过滤。
3. 不为每个外键盲目建立重复索引，但 PostgreSQL 不自动为 FK 建索引，热点 FK 必须显式评估。
4. 部分唯一索引用于“仅活跃记录唯一”或“同时最多一个 OPEN”。
5. JSONB 只在已有明确查询条件时建立 GIN，不预先泛化。

## 2. 主键与组合租户约束

- 所有实体使用 `PRIMARY KEY(id)`。
- 所有租户父表建立 `UNIQUE(workspace_id,id)`，供子表组合 FK 使用。
- 项目级子表使用 `(workspace_id,project_id)` 组合 FK。
- 更深层实体根据父边界使用 `(workspace_id,milestone_id)`、`(workspace_id,receivable_id)` 等组合 FK。
- `ON DELETE CASCADE` 只允许无商业历史的临时/纯技术子记录；商业表默认 `RESTRICT` 或 `NO ACTION`。

## 3. 唯一约束清单

| 表 | 约束 | 目的 |
|---|---|---|

| project | `UNIQUE(workspace_id,id)` | 组合 FK 父键 |
| project | `UNIQUE(workspace_id,project_no)` | 租户内项目编号唯一 |
| milestone | `UNIQUE(workspace_id,id)` | 组合 FK 父键 |

## 4. CHECK 约束清单

| 规则 | 示例 |
|---|---|
| 金额非负 | `amount_due >= 0`, `amount_paid >= 0`, `confirmed_amount >= 0` |
| 付款金额为正 | `payment_record.amount > 0` |
| 币种格式 | `currency ~ '^[A-Z]{3}$'` |
| 日期顺序 | `start_date IS NULL OR target_end_date IS NULL OR start_date <= target_end_date` |
| Milestone 顺序 | `sequence_no >= 1` |
| 归档字段一致 | `status <> 'ARCHIVED' OR archived_at IS NOT NULL` |
| 作废字段一致 | `status <> 'VOIDED' OR (voided_at IS NOT NULL AND void_reason IS NOT NULL)` |
| Actor 一致性 | USER 必须有 actor_user_id；CLIENT 可有 public_link_id |
| Action 金额币种配对 | amount_impact 与 currency 同时为空或同时有值 |
| PublicLink 失效字段 | REVOKED/SUPERSEDED 与对应时间字段一致 |
| 决定原因 | REJECTED/REVISION_REQUESTED 时 reason/message 非空 |
| 应收余额 | 应用层保证 `amount_paid <= amount_due`；若允许超付需显式业务规则，不在未确认前硬编码 |

## 5. 外键策略

### 5.1 必须使用组合 FK 的关系

```text
client(workspace_id,id) <- project(workspace_id,client_id)
project(workspace_id,id) <- milestone/quote/change/receivable/audit/action/feedback
milestone(workspace_id,id) <- delivery_draft/delivery_version/acceptance_decision
receivable(workspace_id,id) <- payment_record/reminder
public_link(workspace_id,id) <- public_session/public_access_log
file_asset(workspace_id,id) <- file_reference
```

### 5.2 删除行为

| 父对象 | 子对象 | ON DELETE |
|---|---|---|
| Workspace | 所有业务对象 | RESTRICT |
| User | Membership/AuthSession | RESTRICT；通过状态撤销 |
| Client | Project | RESTRICT |
| Project | 商业子对象 | RESTRICT |
| 草稿 | 未发布临时子项 | CASCADE 可接受，但需确认无历史引用 |
| 发布版本 | Decision/PublicLink/FileReference | RESTRICT |
| Receivable | Payment/Reminder | RESTRICT |
| EventPublication | EmailTask | RESTRICT 或受控 CASCADE（优先保留审计） |

## 6. 查询索引策略

### 6.1 核心列表

```sql
CREATE INDEX idx_project_workspace_status_updated
  ON project(workspace_id, status, updated_at DESC);

CREATE INDEX idx_project_workspace_client_updated
  ON project(workspace_id, client_id, updated_at DESC)
  WHERE client_id IS NOT NULL;

CREATE INDEX idx_milestone_workspace_project_sequence
  ON milestone(workspace_id, project_id, sequence_no);

CREATE INDEX idx_milestone_workspace_status_due
  ON milestone(workspace_id, status, due_date)
  WHERE status NOT IN ('ACCEPTED','CANCELLED');
```

### 6.2 ActionCenter、Feedback 与时间线

```sql
CREATE INDEX idx_action_workspace_status_priority_due
  ON action_item_projection(workspace_id, status, priority, due_at);

CREATE INDEX idx_action_workspace_project_status
  ON action_item_projection(workspace_id, project_id, status, updated_at DESC);

CREATE INDEX idx_feedback_workspace_project_time
  ON feedback_projection(workspace_id, project_id, occurred_at DESC);

CREATE INDEX idx_audit_workspace_project_time
  ON audit_event(workspace_id, project_id, occurred_at DESC);

CREATE INDEX idx_audit_workspace_aggregate_time
  ON audit_event(workspace_id, aggregate_type, aggregate_id, occurred_at DESC);
```

### 6.3 应收与逾期

```sql
CREATE INDEX idx_receivable_workspace_status_due
  ON receivable(workspace_id, status, due_date)
  WHERE status IN ('OPEN','PARTIALLY_PAID','OVERDUE');

CREATE INDEX idx_payment_workspace_receivable_paid
  ON payment_record(workspace_id, receivable_id, paid_at DESC);
```

### 6.4 公开访问与安全

```sql
CREATE UNIQUE INDEX uk_public_link_token_hash ON public_link(token_hash);
CREATE INDEX idx_public_link_workspace_object
  ON public_link(workspace_id, object_type, object_id, status);
CREATE INDEX idx_public_link_expiry
  ON public_link(status, expires_at)
  WHERE status = 'ACTIVE';
CREATE INDEX idx_auth_session_access_hash ON auth_session(access_token_hash);
CREATE INDEX idx_auth_session_refresh_hash ON auth_session(refresh_token_hash);
CREATE INDEX idx_auth_session_family ON auth_session(session_family_id, status);
```

### 6.5 Worker 与幂等

```sql
CREATE INDEX idx_event_publication_claim
  ON event_publication(status, next_attempt_at, occurred_at)
  WHERE status IN ('PENDING','RETRY');

CREATE INDEX idx_event_publication_lease
  ON event_publication(locked_until)
  WHERE status = 'PROCESSING';

CREATE INDEX idx_email_task_claim
  ON email_task(status, next_attempt_at)
  WHERE status IN ('PENDING','RETRY');

CREATE INDEX idx_idempotency_expiry
  ON idempotency_record(expires_at);
```

## 7. 不建议建立的索引

- 低基数且不带 Workspace 的单列 `status` 索引。
- 未有明确查询的所有 JSONB GIN 索引。
- 与唯一约束完全重复的普通索引。
- 只为排序但没有过滤前缀的时间索引。
- 对可重建投影建立过多写放大索引；以实际 EXPLAIN 为依据。

## 8. 不可变与审计约束

1. 发布版本正文列由触发器拒绝 UPDATE/DELETE，或拆分为 immutable content 表与 mutable lifecycle 表。
2. `audit_event`、`public_access_log`、`email_attempt` 仅追加。
3. PaymentRecord 通过 VOIDED 补偿，不物理删除。
4. FileAsset 存在 FileReference 时禁止物理删除。
5. 所有触发器和权限在 Testcontainers PostgreSQL 集成测试中验证。

## 9. 索引评审与监控

- 每次新增索引必须给出目标 SQL、预期选择性和写入成本。
- Staging 使用接近生产的数据量执行 `EXPLAIN (ANALYZE, BUFFERS)`。
- 初始慢查询阈值 300ms，记录 SQL 指纹，不记录敏感参数。
- 定期检查未使用索引、重复索引、索引膨胀、锁等待和顺序扫描。
- 大表索引在生产使用 `CREATE INDEX CONCURRENTLY`，并由独立非事务 Flyway 迁移执行。
