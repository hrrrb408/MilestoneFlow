# 《MilestoneFlow Pilot MVP V0.1 数据迁移规范》

## 1. 目标

- 使用 Flyway 对数据库结构、约束、索引、触发器和必要种子数据进行版本化。
- 迁移只追加，不修改已经在共享环境执行过的版本。
- 生产采用 Expand → Migrate → Contract，应用回滚不依赖破坏性 Down Migration。
- 每次发布保持至少“当前应用版本与上一应用版本”对数据库结构兼容。


规则：

- 版本号单调递增，不按开发者分支重用。
- 文件名使用英文 snake_case，描述清晰。
- `V*` 迁移永不修改；`R*` 仅用于可重复创建的 View/Function，不用于不可逆业务数据变更。
- 一次迁移只做一个可评审主题，避免数千行万能脚本。

## 4. Expand–Migrate–Contract 操作规范

### 4.1 新增非空字段示例

错误做法：在大表直接 `ADD COLUMN xxx NOT NULL DEFAULT ...` 并立即要求新应用写入。

正确步骤：

1. **Expand：** 新增 nullable 字段或安全默认值，旧应用继续运行。
2. **Deploy Compatible App：** 新应用同时兼容旧字段与新字段，并开始双写或优先写新字段。
3. **Migrate：** 通过可重入批处理回填历史数据。
4. **Verify：** 校验空值数量、抽样业务一致性和性能。
5. **Contract：** 后续独立发布添加 NOT NULL/CHECK，停止旧字段写入。
6. **Cleanup：** 再后续发布删除旧字段。

### 4.2 字段重命名

不直接 rename 后立即发布。先新增新列并双写，回填后切换读取，最后删除旧列。API 与数据库命名变更分别管理。

### 4.3 枚举新增

1. 先扩展数据库 CHECK 允许新值。
2. 发布能够读取但不一定写入新值的兼容应用。
3. 发布开始写新值的业务版本。
4. 旧实例退出后再考虑收紧旧值。

### 4.4 大表索引

- 生产使用 `CREATE INDEX CONCURRENTLY`。
- PostgreSQL 并发索引不能在事务块内执行，应使用专用 Flyway 配置/迁移并明确失败恢复步骤。
- 迁移前检查磁盘空间、锁和预计时长。

## 5. 数据回填规范

- 回填程序必须可重入、可暂停、可按主键/时间窗口分页。
- 每批提交，避免长事务和大量行锁。
- 记录批次、开始/结束位置、成功/失败数和最后错误。
- 业务时间计算必须显式使用 Workspace IANA 时区。
- 金额迁移使用 BigDecimal/decimal，不经过浮点。
- 回填完成后执行总数、空值、金额合计、状态分布和外键孤儿校验。

## 6. 事务与锁

- 默认每个 Flyway 版本在事务中执行；不支持事务的 DDL 单独迁移。
- 禁止在高峰期执行需要长时间 ACCESS EXCLUSIVE 锁的 DDL。
- 对大表添加约束可先 `NOT VALID`，后台验证后再 `VALIDATE CONSTRAINT`。
- 迁移必须设置合理的 `lock_timeout` 和 `statement_timeout`，失败后停止发布，不无限等待。

## 7. 发布流程

1. CI 在空 PostgreSQL 17 数据库执行全量迁移。
2. CI 从“上一正式版本数据库快照”执行增量迁移。
3. 启动后端，Hibernate 使用 `ddl-auto=validate` 验证 ORM。
4. 执行约束、触发器、组合 FK、租户隔离和关键查询集成测试。
5. Staging 使用生产同主版本数据库演练，记录时长和锁影响。
6. 生产发布前创建可恢复点并确认 PITR/备份可用。
7. 显式执行 Flyway migrate；失败则不启动新应用。
8. 发布兼容应用并执行 Smoke Test。
9. 监控错误率、锁、慢查询、Worker 队列和业务对账。

## 8. 回滚策略

- 应用回滚：切回上一不可变镜像，但数据库必须保持向后兼容。
- 数据库不以 Down Migration 为主要方案。
- 数据修复优先前滚补偿迁移。
- 若迁移已造成不可接受的数据破坏，冻结写入并使用 PITR/备份恢复，再执行事件与对象引用对账。
- 破坏性 Contract 迁移必须延迟到新版本稳定后，且单独审批。

## 9. 种子数据与参考数据

- V0.1 枚举主要由代码和 CHECK 管理，不建立可随意修改的通用 enum 表。
- 必需系统模板、错误码映射或邮件模板版本可通过独立版本化表和迁移插入。
- 种子 INSERT 必须幂等或使用确定主键，并明确是否允许后续业务修改。
- 不在迁移脚本中写入真实用户、客户、Token 或生产密钥。

## 10. 数据校验 SQL 基线

发布后至少检查：

```sql
-- 跨租户项目/里程碑孤儿应为 0
SELECT count(*)
FROM milestone m
LEFT JOIN project p
  ON p.workspace_id = m.workspace_id AND p.id = m.project_id
WHERE p.id IS NULL;

-- 同一发布版本多重决定应为 0
SELECT quote_version_id, count(*)
FROM quote_decision
GROUP BY quote_version_id
HAVING count(*) > 1;

-- 应收金额非法应为 0
SELECT count(*)
FROM receivable
WHERE amount_due < 0 OR amount_paid < 0;

-- 活跃工作空间多 Owner 应为 0
SELECT workspace_id, count(*)
FROM workspace_membership
WHERE role='OWNER' AND status='ACTIVE'
GROUP BY workspace_id
HAVING count(*) > 1;
```

## 11. 测试门禁

- 空库迁移成功。
- 上一版本增量迁移成功。
- 重复执行可重复迁移无差异。
- 所有 FK、唯一、CHECK、部分索引和触发器存在。
- 跨 Workspace 写入被数据库拒绝。
- 发布版本 UPDATE/DELETE 被拒绝，但合法撤销/替代状态变化可执行。
- 50 并发确认只产生一个基线/最终结论/应收副作用。
- Worker `FOR UPDATE SKIP LOCKED` 不重复领取。
- 回填程序中断后可恢复且不会重复副作用。

## 12. 变更审批清单

每个数据库变更 PR 必须说明：

- 业务需求与关联实体；
- DDL/DML 内容；
- 向前/向后兼容性；
- 预计数据量、锁级别和耗时；
- 新增/删除索引及写入成本；
- 回填与校验方案；
- 应用回滚行为；
- 备份/PITR 需求；
- 测试证据与 Staging 演练结果。
