## 2. 建模目标与原则

1. 每条内部业务数据必须归属于唯一工作空间，除全局身份认证数据外，业务表必须具有 `workspace_id NOT NULL`。
2. 聚合边界围绕业务一致性定义，不按页面或数据库表机械划分。
3. 报价、交付、变更采用“可编辑草稿 + 不可变发布版本”；历史商业事实只追加，不覆盖。
4. `Task` 在 V0.1 定义为由业务状态派生的 `ActionItemProjection`，不提供通用 Todo CRUD。
5. `Feedback` 在 V0.1 定义为领域原始反馈的统一只读投影，不形成通用评论或聊天系统。
6. 金额使用 `BigDecimal / numeric(19,4)`；技术时间使用 UTC `timestamptz`；业务日期使用 `date` 并由工作空间时区解释。
7. 聚合之间只传 ID、值对象、应用 DTO 或领域事件，不共享 JPA Entity 与 Repository。
8. 软删除只用于允许恢复的可变主数据；商业历史使用归档、撤销、作废、替代等显式状态。

## 3. 统一业务术语表

| 中文术语 | 英文术语 | 定义 | 不包含 |
|---|---|---|---|
| 用户 | User | 可登录 MilestoneFlow 的内部账号，V0.1 主要为 Owner | 外部客户联系人 |
| 工作空间 | Workspace | 数据隔离、币种、时区和权限的租户边界 | 多组织集团管理 |
| 工作空间成员 | WorkspaceMember | User 与 Workspace 的角色关系；V0.1 仅开放 OWNER | 将角色直接放入 User |
| 客户 | Client | 接受报价、交付或变更确认的外部商业主体 | 系统登录账号 |
| 项目 | Project | 一次固定价商业交付的生命周期容器 | 通用工时或敏捷迭代 |
| 商业基线 | CommercialBaseline | 已确认报价及后续已确认变更形成的不可变商业事实快照 | 前端动态拼接结果 |
| 里程碑 | Milestone | 项目中的可交付、可验收阶段 | 通用任务列表 |
| 交付版本 | DeliveryVersion | 一次正式提交给客户验收的不可变交付快照 | 可覆盖编辑的附件集合 |
| 行动项/任务 | ActionItem / Task | 由报价、交付、变更、应收等状态派生的下一动作 | 用户自由创建 Todo |
| 活动日志 | ActivityLog | 关键状态、操作者、来源和请求编号的不可变审计事件 | 可编辑动态或聊天消息 |
| 反馈 | Feedback | 报价留言、拒绝原因、交付修改请求、变更留言等领域反馈的统一只读视图 | 通用评论线程 |
| 应收 | Receivable | 由确认报价、验收或确认变更产生的应收事实 | 在线支付资金账户 |
| 公开链接 | PublicLink | 绑定单个已发布业务对象和允许动作的能力令牌记录 | 登录账号或全项目访问权 |
| 发布版本 | PublishedVersion | 已对外发布、正文不可更新或删除的业务版本 | 草稿状态记录 |
| 归档 | Archive | 业务对象进入只读保留状态 | 物理删除历史 |
| 作废 | Void | 保留原记录并使其商业效果失效 | 删除付款记录 |
| 替代 | Supersede | 新版本发布后旧待处理版本不再允许客户操作 | 修改旧版本正文 |

## 5. 核心实体定义

### 5.1 User

**职责：** 表示内部登录主体，不直接承载工作空间角色。

**关键属性：** `id`、规范化邮箱、显示名称、密码哈希、验证时间、账号状态、语言、最后登录时间。

**业务规则：**

- `email_normalized` 全局唯一，保存前执行 trim、Unicode 规范化和小写化。
- 密码哈希、验证令牌和会话令牌不得写入 ActivityLog。
- 禁用、密码重置或全端退出必须撤销相关 AuthSession。
- User 与 Workspace 通过 WorkspaceMember 建立关系。

### 5.2 Workspace

**职责：** 作为所有内部业务数据的租户边界，并保存默认币种和 IANA 时区。

**业务规则：**

- V0.1 一个工作空间只能有一个活跃 OWNER。
- 工作空间归档后禁止创建或修改商业对象，但历史查询、审计和导出仍可用。
- 默认币种或时区变更不改写历史记录；历史业务对象保留自身币种和业务日期解释。

### 5.3 WorkspaceMember

**职责：** 表示 User 在 Workspace 中的成员资格与角色。

**业务规则：**

- V0.1 仅启用 `OWNER`，但模型预留 `ADMIN/MEMBER/VIEWER`。
- 同一用户在 V0.1 只允许一个活跃成员关系。
- 移除成员通过状态和 `ended_at` 表达，不物理删除。

### 5.4 Project

**职责：** 维护固定价项目生命周期、归档状态以及当前商业基线引用。

**业务规则：**

- Project 必须属于一个 Workspace，可关联一个 Client。
- 币种在项目创建后原则上不可变；需要变更时通过受控迁移或新项目处理。
- 项目进入 `ARCHIVED` 后所有业务写入由统一 Guard 拒绝。
- `current_baseline_id` 只能指向同一 Workspace 和 Project 的 CommercialBaseline。

### 5.5 Milestone

**职责：** 表示项目中可独立交付和验收的阶段。

**业务规则：**

- 只有项目存在当前商业基线时才能进入可执行状态。
- 同一项目内 `sequence_no` 唯一，用于稳定排序。
- 已产生交付版本的里程碑不可物理删除，只能取消或保留。
- 验收通过后记录 `accepted_at`，不允许回到编辑态。

### 5.6 Task（ActionItemProjection）

**职责：** 聚合显示待发送、待确认、待验收、待付款、已逾期和邮件失败等下一动作。

**业务规则：**

- 只由领域事件或投影重建任务生成，不允许用户任意创建。
- `source_type + source_id + action_type` 标识来源；同一来源同一动作同时最多一条 OPEN 记录。
- 源对象状态变化后关闭、替代或重新生成行动项。
- 投影数据允许重建，因此不作为商业事实的唯一来源。

### 5.7 ActivityLog（物理表 `audit_event`）

**职责：** 保存业务关键动作、操作者、来源、请求编号和脱敏元数据。

**业务规则：**

- 仅 INSERT；普通业务 API 不提供 UPDATE/DELETE。
- 允许 `USER`、`CLIENT`、`SYSTEM`、`JOB` 四种 Actor。
- `request_id` 用于关联 API、日志、Trace 和幂等记录。
- `metadata` 只保存白名单 JSON，不保存令牌、密码、完整 Cookie、文件正文或客户敏感全文。

### 5.8 Feedback（物理表 `feedback_projection`）

**职责：** 统一展示报价、交付和变更中的反馈。

**业务规则：**

- 原始反馈仍存放在 QuoteDecision、AcceptanceDecision、ChangeDecision 等所属领域记录中。
- Feedback 是只读投影，不允许直接修改原始内容。
- 投影可从源记录和领域事件重建。
- 同一源反馈在投影中保持唯一，避免重复展示。

## 6. 关键值对象

| 值对象 | 字段 | 规则 |
|---|---|---|
| WorkspaceId / ProjectId 等 | UUID | Java 类型区分，数据库统一 `uuid` |
| Money | amount + currency | `amount numeric(19,4)`；`currency char(3)`；同一运算币种必须一致 |
| BusinessDate | LocalDate + Workspace Zone | 数据库存 `date`；按 Workspace IANA 时区解释 |
| EmailAddress | original + normalized | 显示值与规范化值分离；唯一约束使用 normalized |
| VersionNumber | positive integer | 报价、交付、变更发布版本按父对象递增，起始为 1 |
| Actor | actorType + actorId | Client/System 可没有 userId，但必须能解析来源 |
| PublicCapability | objectType + objectId + allowedActions | 单一业务对象绑定，不允许扩大权限集合 |
| RequestFingerprint | operation + idempotencyKey + requestHash | 同 Key 不同请求哈希必须冲突 |

## 7. 跨聚合事务边界

### 7.1 报价确认

同一事务内：创建 QuoteDecision → 创建初始 CommercialBaseline → 创建应收草稿 → 更新 Project 汇总/状态 → 写 ActivityLog → 写 EventPublication。

### 7.2 交付验收通过

同一事务内：创建 AcceptanceDecision → 更新 Milestone 状态 → 激活或创建 Receivable → 写 ActivityLog → 写 EventPublication。

### 7.3 变更确认

同一事务内：校验 `base_baseline_id` 仍为当前基线 → 创建 ChangeDecision → 生成下一版 CommercialBaseline → 更新 Project 当前基线和汇总 → 创建/调整 Receivable → 写 ActivityLog 与 EventPublication。

### 7.4 付款记录或作废

同一事务内：写 PaymentRecord 或 Void 信息 → 重算 Receivable 已收/未收和状态 → 写 ActivityLog → 写 EventPublication。幂等记录与业务副作用同事务提交。

## 8. 工作空间数据隔离模型

### 8.1 字段规则

- 全局表：`app_user`、部分认证令牌表，不含 `workspace_id`。
- 租户业务表：必须含 `workspace_id uuid NOT NULL`。
- 项目级子表：同时包含 `workspace_id` 和 `project_id`。
- 审计、文件、公开链接、幂等记录、异步事件也必须保留 `workspace_id`。

### 8.2 组合外键规则

父表除主键外建立 `UNIQUE(workspace_id, id)`；子表使用：

```sql
FOREIGN KEY (workspace_id, project_id)
REFERENCES project (workspace_id, id)
```

更深层级同理，例如 DeliveryVersion 通过 `(workspace_id, milestone_id)` 引用 Milestone。即使 UUID 全局唯一，也保留组合约束作为租户写入的数据库防线。

### 8.3 查询规则

Repository 不暴露 `findById(id)` 类型业务查询，只允许 `findByWorkspaceIdAndId(workspaceId, id)` 或明确的全局认证查询。不存在和跨租户统一返回不可见。

## 9. 审计字段与软删除策略

### 9.1 可变聚合通用字段

```text
id uuid
created_at timestamptz
created_by uuid nullable
updated_at timestamptz
updated_by uuid nullable
version bigint
```

`version` 用于 JPA 乐观锁。`created_by/updated_by` 对 System 或 Client 动作可为空，具体来源写入 ActivityLog。

### 9.2 删除策略矩阵

| 数据类型 | 策略 | 字段/状态 |
|---|---|---|
| User | 禁用为主；合规删除另行流程 | `status=DISABLED`，必要时匿名化 |
| Workspace | 归档 | `status=ARCHIVED`、`archived_at/by` |
| WorkspaceMember | 终止成员关系 | `status=REMOVED`、`ended_at` |
| Client | 软归档 | `status=ARCHIVED`、`archived_at/by` |
| Project | 归档只读 | `status=ARCHIVED`、`archived_at/by` |
| Milestone | 无历史时允许删除；有历史时取消 | `status=CANCELLED` |
| 草稿 | 未发布且无引用时可硬删除，建议仍用取消状态 | `status=CANCELLED` 或 `deleted_at` |
| 发布版本/决策/基线/付款 | 禁止删除 | 撤销、替代、作废等显式状态 |
| ActivityLog | 不可改删 | INSERT ONLY |
| Task/Feedback 投影 | 可重建 | 允许批量清理后重建，不作商业审计来源 |

不使用 Hibernate 全局 `@SQLDelete/@Where` 隐藏过滤，以免后台任务、审计和跨状态查询出现不可见副作用。

## 10. ORM 映射建议（待后端窗口确认）

| 主题 | 建议基线 |
|---|---|
| ORM | Spring Data JPA / Hibernate；复杂列表与工作台使用 Projection 或显式 SQL |
| ID | Java `UUID`，由应用生成；数据库类型 `uuid` |
| 枚举 | `@Enumerated(EnumType.STRING)` + 数据库 CHECK；不使用 PostgreSQL 原生 ENUM |
| 金额 | `@Embeddable Money(BigDecimal amount, String currency)`，precision=19、scale=4 |
| 时间 | `Instant` ↔ `timestamptz`；`LocalDate` ↔ `date`；时区保存 IANA 字符串 |
| 乐观锁 | 可变聚合根使用 `@Version long version` |
| 关联 | 默认保存外键 ID；仅聚合内部使用受控 `@OneToMany`；禁止跨模块大对象图和 EAGER |
| workspace | 实体显式持有 `workspaceId`；Repository 方法必须以 workspaceId 为首要条件 |
| JSONB | 仅用于设置、脱敏元数据、响应快照；使用 Hibernate JSON 类型映射 |
| OSIV | 关闭 `spring.jpa.open-in-view` |
| 删除 | 显式领域状态与 Repository 条件，不依赖全局软删除注解 |
| 不可变表 | `@Immutable` 可作为 ORM 提示，数据库触发器/权限作为最终防线 |

**后端窗口确认项：** 包结构、ID 生成策略、Money Embeddable、JSONB 映射方式、审计字段自动填充、组合外键映射、Projection SQL 技术（JPA Projection/JdbcClient/jOOQ）以及 Flyway 与 JPA `ddl-auto=validate` 的启动顺序。

## 11. 领域设计验收标准

- 八个指定核心实体均有职责、边界和持久化映射。
- Task 与 Feedback 不突破冻结范围。
- 所有项目级子实体可通过组合外键阻止跨 Workspace 引用。
- 每个状态机有合法/非法转换测试。
- 已发布版本、基线、决策、付款和 ActivityLog 不可被普通业务接口删除。
- 跨聚合强一致流程在单事务失败时整体回滚。
- ORM 映射经后端窗口评审后，差异以 ADR 或变更记录回写本文件。
